Система управления пользователями и контентом с кастомной реализацией RBAC (Role-Based Access Control) на FastAPI.

## Описание проекта

Проект представляет собой API для управления пользователями, продуктами и статьями. Главная особенность — гибкая система прав, хранящаяся в базе данных, которая разделяет доступ к "своим" ресурсам и "всем" ресурсам в системе.

## Система ролей и разрешений

Права доступа проверяются через таблицу `permissions` с использованием следующих флагов:

* **{action}_perm**: Разрешает действие над ресурсом, владельцем которого является текущий пользователь (`resource.owner_id == user.id`).
* **{action}_all_perm**: Разрешает действие над ресурсом любого пользователя (административный доступ).

### Матрица прав

1. **Admin (Роль 1)**
* Доступ ко всем ресурсам (Product, Article, User) на чтение, создание, обновление и удаление всех записей.
* Возможность назначать `owner_id` вручную при создании контента.
* Возможность изменять `role_id` любого пользователя.


2. **Creator (Роль 2)**
* Продукты/Статьи: создание, просмотр всех, редактирование и удаление **только своих**.
* Профиль: редактирование и мягкое удаление **своего** аккаунта.


3. **User (Роль 3)**
* Продукты/Статьи: только просмотр всех записей.
* Профиль: редактирование и мягкое удаление **своего** аккаунта.

## Основные эндпоинты

### Аутентификация

* **POST /auth/register**
* Регистрирует нового пользователя. По умолчанию всегда назначается `role_id: 3` (User), даже если в теле запроса передано другое значение.


* **POST /auth/login**
* Авторизация по email и паролю.
* **Ответ**: access token и refresh token



### Пользователи (Users)

* **GET /users**
* Список всех пользователей.


* **GET /users/{id}**
* Получение данных конкретного пользователя.


* **PATCH /users/{id}**
* Обновление профиля. Обычный пользователь может менять только свои данные (имя, фамилия). Менять `role_id` может только Admin.


* **DELETE /users/{id}**
* Мягкое удаление аккаунта. Устанавливает `is_active = False`. Пользователь больше не может войти в систему.



### Продукты (Products)

* **GET /products**
* Просмотр списка всех продуктов.


* **POST /products**
* Создание нового продукта. `owner_id` подставляется автоматически из ID текущего пользователя. Admin может переопределить это поле в запросе.


* **PATCH /products/{id}**
* Частичное обновление продукта.


* **DELETE /products/{id}**
* Удаление продукта. 



### Статьи (Articles)

* **GET /articles**
* Просмотр всех статей.


* **POST /articles**
* Создание статьи. Логика `owner_id` идентична продуктам.


* **PATCH /articles/{id}**
* Редактирование статьи (только автор или админ).


* **DELETE /articles/{id}**
* Удаление статьи (только автор или админ).

## Технические детали

### Безопасность и проверка прав

Для защиты эндпоинтов используется зависимость `Checker`, которая выполняет следующие действия:

1. Идентифицирует пользователя по JWT.
2. Проверяет статус `is_active`.
3. Запрашивает права роли из БД для текущего ресурса и действия.
4. Если запрашивается конкретный ID (`/resource/{id}`), проверяет, является ли пользователь владельцем (поле `owner_id`), если у него нет "global" прав (`all_perm`).

### Обработка исключений

Система использует кастомные исключения:

* **401 Unauthorized**: Ошибка токена или неверный логин.
* **403 Forbidden**: Недостаточно прав для действия или попытка изменить чужой ресурс.
* **404 Not Found**: Ресурс не найден или пользователь деактивирован.

## Установка и запуск

1. Подготовьте среду .env
2. Запустите контейнеры:
```bash
docker-compose up --build
```

3. Выполните миграции:
```bash
docker-compose exec app alembic upgrade head
```
